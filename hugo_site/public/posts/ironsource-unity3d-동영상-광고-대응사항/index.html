향후 변경 사항#

아이언소스 최신 버전 업데이트 관련한 사항입니다. 참조 부탁 드립니다.

8.11.1에서 JDK 버전이 변경된 이유는 게임 제작에 사용하는 유니티 에디터의 구 버전에서 레벨플레이 유니티 플러그인 사용 시 이슈가 발생하여 이를 방지하기 위해 나온 버전입니다. 해당 사항 이외에는 변경점이 없으며, 이미 8.11 및 JDK 17로 빌드를 하셨다면 교체하지 않으셔도 무방합니다.

9.0.0은 이전 말씀드린 바와 같이 기존에 사용했던 IronSource API가 대부분 폐기되며 LevelPlay API (광고 유닛  API) 사용이 강제됩니다. target SDK 35는 8.11.0 이후 버전에서는 모두 대응되며, SDK 자체도 API level 35로 빌드되니 참고 부탁 드립니다.

  • 8.11.0 은 OS ==Target 15 대응== 이라 기존 예외 처리 제거 필요
    • IronSource 8.11.0 버전으로 TargetAPI35(OS15) 대응 완료로, Edge2Edge 예외처리 삭제 
    • Ref: https://jira.skplanet.com/browse/OCBNATIVE-853
  • 동영상 광고 SDK 와 Proximity SDK 내부 SDK 충돌관련 Proximity sdk 버전 변경 - https://jira.skplanet.com/browse/OCBNATIVE-888
    • Protobuffer 라이브러리 충돌로 인한 변경

JS 호출건 관련#

9.0.0 이상으로 변경되었을시 하위 버전의 api 모두 LevelPlay API 사용으로 강제 됩니다. 이럴 경우 기존 보상형동영상 광고의 경우 placementName 기반으로 광고 단위를 구분했었는데 유닛아이디(unit_id) 기반으로 변경됩니다.

현재 Syrup의 경우 보상형 동영상광고의 경우 placementName을 JS 호출하여 사용하였으나 unit_id 방식으로 변경해야 하기때문에 ==모웹에서 버전 분기== 필요 합니다. 다만 기존 Interstitial 비보상형 광고는 unit_id 사용 합니다. 추가로 고려 해야할 사항으로 게임참여적립 (tapjoy offerall sdk)의 경우도 placementName을 사용하고 있는데 이 부분도 영향도가 있는지 검토 필요 합니다.

위와 관련된 JS 규격은 아래 와 같습니다.

  • requestIronSourceAdWindow (보상형 동영상광고)
  • requestOfferwall (게임참여 적립)

[!info] Info setupLevelPlayListeners(adUnitId: String) 를 사용하면 되는가? 어드민을 통한 미디에이션을 사용하는 방식으로 기획팀은 고려 중

기존 requestIronSourceAdWindow 호출 구조#
  • partnerServiceId 정보를 통해 5420 전문으로 user 정보 받아옴 , placementName (광고 단위), tabeventAdKey 정보를 받아 app i/f 적립 할지 여부 결정
  • 광고의 종류는 IronSource 보상형동영상광고
  • tabeventAdKey 가 없을 시
    • JS > Ironsource sdk > 광고 호출 및 로드 > 광고 리워드 완료
  • tabeventAdKey 가 있을 시
    • JS > Ironsource sdk > 광고 호출 및 로드 > App I/F

모웹 : 보상형 동영상 광고 호출 (JS : requestIronSourceAdWindow) 단말 : ironSource SDK 초기화가 아직 안되어있으면 5420 전문 호출 요청 하여 userId 정보 요청 (App I/F) App I/F : 서비스에 해당 하는 user Id 단말에 전송 ironSource SDK 초기화 광고 load 요청 광고 show (placementName) tabeventAdKey 값이 있다면 광고 시청 후 app I/F 에 적립 요청 후 적립 후 모웹에 콜백 리턴 tabeventAdKey 값이 없다면 광고 시청 후 닫힐때 모웹에 콜백 제공 모웹 : 광고 결과 콜백 후 금 적립 요청

타 SDK와 버전 충돌 관련#

기존 포인트홈(구 캐시버튼) sdk 의 경우 내부적으로 ironSource 관련된 광고를 사용하고 있는 것으로 보여지는데 해당 sdk와 충동 이슈 없는지 고려 필요 합니다.
